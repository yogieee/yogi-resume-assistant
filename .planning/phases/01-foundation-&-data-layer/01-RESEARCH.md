# Phase 1: Foundation & Data Layer - Research

**Researched:** 2026-03-11
**Domain:** Next.js 16 project scaffold, TypeScript data layer, Tailwind CSS v4 dark theme
**Confidence:** HIGH

## Summary

Phase 1 establishes the project scaffold (Next.js 16 + TypeScript + Tailwind CSS v4), defines the typed portfolio data layer as a single source of truth, and implements the dark console theme with monospace fonts and glowing accents. This phase has no dependencies and produces the foundation consumed by all downstream phases.

The standard approach is: scaffold with `create-next-app`, configure `next/font` for JetBrains Mono, define theme tokens via Tailwind v4's CSS-based `@theme` directive (no tailwind.config.js), create a single `portfolio.ts` data file with comprehensive TypeScript interfaces, and set up the layout shell with dark background and CSS custom properties for glow colors.

**Primary recommendation:** Use `create-next-app` defaults (TypeScript, Tailwind, ESLint, App Router, Turbopack), configure shadcn/ui immediately after scaffold, define all theme variables in `globals.css` via `@theme`, and create a single exhaustive `portfolio.ts` file -- not split per section -- since the total data is ~200 lines.

### Pattern 1: CSS-First Theme Configuration (Tailwind v4)

**What:** Define all design tokens (colors, fonts, shadows) in `globals.css` using `@theme`, not in a JavaScript config file.

**When to use:** Always in Tailwind v4 projects. The JS config (`tailwind.config.js`) is legacy.

**Example:**

```css
/* src/app/globals.css */
@import "tailwindcss";
@import "tw-animate-css";

/* Custom dark theme variant for potential future use */
@custom-variant dark (&:where(.dark, .dark *));

@theme {
  /* Console accent colors */
  --color-glow-cyan: oklch(0.82 0.15 192);
  --color-glow-green: oklch(0.78 0.18 142);
  --color-glow-purple: oklch(0.65 0.2 290);
  --color-glow-amber: oklch(0.8 0.15 75);

  /* Background colors */
  --color-console-bg: #0a0a0f;
  --color-console-surface: #12121a;
  --color-console-border: #1e1e2e;

  /* Text colors */
  --color-console-text: #e2e8f0;
  --color-console-text-dim: #64748b;
  --color-console-text-bright: #f8fafc;

  /* Font families - override defaults */
  --font-sans: "Geist", ui-sans-serif, system-ui, sans-serif;
  --font-mono: "JetBrains Mono", ui-monospace, "Cascadia Code", "Fira Code", Menlo, Monaco, monospace;

  /* Glow shadows for console aesthetic */
  --shadow-glow-sm: 0 0 8px oklch(0.82 0.15 192 / 0.3);
  --shadow-glow-md: 0 0 16px oklch(0.82 0.15 192 / 0.4);
  --shadow-glow-lg: 0 0 24px oklch(0.82 0.15 192 / 0.5), 0 0 48px oklch(0.82 0.15 192 / 0.2);
}

/* Base styles for dark console aesthetic */
@layer base {
  html {
    color-scheme: dark;
  }
  body {
    @apply bg-console-bg text-console-text font-mono antialiased;
  }
}
```

### Pattern 2: Font Loading with next/font + Tailwind v4

**What:** Use `next/font` to self-host fonts (performance + privacy), expose as CSS variables, wire to Tailwind via `@theme`.

**When to use:** Always. Google Font `@import` in CSS works but `next/font` is the Next.js standard for zero-layout-shift font loading.

**Example:**

```typescript
// src/app/layout.tsx
import { JetBrains_Mono } from "next/font/google";
import { Geist } from "next/font/google";
import "./globals.css";

const jetbrainsMono = JetBrains_Mono({
  subsets: ["latin"],
  variable: "--font-mono",
  display: "swap",
});

const geist = Geist({
  subsets: ["latin"],
  variable: "--font-sans",
  display: "swap",
});

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en" className={`dark ${jetbrainsMono.variable} ${geist.variable}`}>
      <body>{children}</body>
    </html>
  );
}
```

**Important:** The CSS variable names from `next/font` (`--font-mono`, `--font-sans`) must match the `@theme` variable names. When `next/font` sets `--font-mono` on `<html>`, Tailwind's `font-mono` utility picks it up automatically because `@theme { --font-mono: ... }` defines the fallback stack, and the CSS variable from `next/font` overrides it at runtime.

### Anti-Patterns to Avoid

- **tailwind.config.js for theming:** Tailwind v4 uses CSS-based `@theme`. Do not create a JS config file.
- **next-themes for dark mode:** The site is dark-only. Hardcode `className="dark"` on `<html>` and set `color-scheme: dark` in CSS. No theme switching infrastructure needed.
- **Split data files per section:** For ~200 lines of data, splitting into 7 files adds import complexity with no benefit. Use one file.
- **Google Font @import in CSS:** Use `next/font` instead. It self-hosts fonts, eliminates render-blocking requests, and prevents layout shift.
- **Defining colors as hex in utilities:** Define all colors as `@theme` variables so they generate utility classes (`text-glow-cyan`, `bg-console-bg`). Don't litter arbitrary hex values in components.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CSS class merging | String concatenation for conditional classes | `cn()` from shadcn/ui (clsx + tailwind-merge) | Handles Tailwind class conflicts (e.g., `px-4` vs `px-2`), conditional classes, and undefined values correctly |
| Font loading | `@import url()` in CSS or `<link>` in head | `next/font/google` | Self-hosts fonts, zero layout shift, automatic subset optimization, no external requests |
| Dark mode infrastructure | Theme provider, context, toggle logic | Hardcode `class="dark"` on `<html>` + `color-scheme: dark` | Site is dark-only. Zero JS needed for theme. |
| CSS reset / normalization | Custom reset CSS | Tailwind's `@layer base` preflight | Tailwind includes Preflight (modern-normalize) automatically |
| Component variant styling | Manual conditional class strings | `class-variance-authority` (via shadcn/ui) | Type-safe variant definitions with autocomplete |

**Key insight:** Phase 1 is all configuration and data. Every "build" task is really a "configure correctly" task. The tools do the work -- you wire them together.

## Common Pitfalls

### Pitfall 1: Tailwind v4 Config Confusion

**What goes wrong:** Developer creates `tailwind.config.js` or `tailwind.config.ts` because all tutorials from 2024 and earlier show this. Tailwind v4 ignores it silently, and custom theme values don't work.
**Why it happens:** Muscle memory from Tailwind v3. Most online examples are v3.
**How to avoid:** All customization goes in `globals.css` via `@theme` directive. Delete any `tailwind.config.*` file if it exists. The only Tailwind-related config file is `postcss.config.mjs` (generated by create-next-app).
**Warning signs:** Custom colors/fonts not generating utility classes. `tailwind.config.js` file exists in the project.

### Pitfall 2: Font Variable Name Mismatch

**What goes wrong:** `next/font` defines `variable: "--font-jetbrains"` but Tailwind's `@theme` expects `--font-mono`. The `font-mono` utility class doesn't use the custom font.
**Why it happens:** The CSS variable name from `next/font` must exactly match the Tailwind theme variable name for the override to work.
**How to avoid:** Use `variable: "--font-mono"` in the `next/font` call so it matches `--font-mono` in `@theme`. Same for sans: use `variable: "--font-sans"`.
**Warning signs:** `font-mono` class shows system monospace fonts instead of JetBrains Mono.

### Pitfall 3: Missing `dark` Class on HTML Element

**What goes wrong:** Dark theme colors defined with `dark:` prefix don't apply. The page renders with light defaults.
**Why it happens:** Tailwind's dark mode requires `class="dark"` on the `<html>` element (when using class strategy, which is the default).
**How to avoid:** Since this is a dark-only site, add `className="dark"` to the `<html>` element in `layout.tsx`. Also set `color-scheme: dark` in CSS to inform the browser (affects scrollbar colors, form controls, etc.).
**Warning signs:** Page background is white. Browser default form elements appear in light mode.

### Pitfall 4: Portfolio Data Types Are Too Loose

**What goes wrong:** Data is typed as `Record<string, any>` or uses string unions that don't match the actual data. Downstream components (command parser, profile card) can't safely access nested fields.
**Why it happens:** Rushing to "just get the data in" without designing interfaces first.
**How to avoid:** Define TypeScript interfaces FIRST in `types/portfolio.ts`, THEN fill in the data in `data/portfolio.ts`. Use `satisfies PortfolioData` to get both inference and validation.
**Warning signs:** `any` type anywhere in the data layer. Components using type assertions (`as`) to access data fields.

### Pitfall 5: shadcn/ui Init Fails with Tailwind v4

**What goes wrong:** Running `npx shadcn@latest init` fails or generates incompatible output because it doesn't detect Tailwind v4 correctly.
**Why it happens:** Older shadcn/ui CLI versions or edge cases in detection.
**How to avoid:** Run `npx shadcn@latest init` (always use `@latest`). The current CLI auto-detects Tailwind v4 and generates compatible CSS using `@theme inline` or OKLCH color variables. If it asks about Tailwind version, explicitly choose v4.
**Warning signs:** Generated `globals.css` contains `@plugin 'tailwindcss-animate'` (v3 pattern) instead of `@import "tw-animate-css"` (v4 pattern).

### Pitfall 6: Not Committing the Data With All Sections

**What goes wrong:** Developer creates portfolio data with some sections filled and others as `TODO` or empty arrays. Downstream phases discover missing data and have to backfill.
**Why it happens:** Data entry feels tedious. Developer wants to "move on to the fun stuff."
**How to avoid:** Phase 1 success criteria explicitly requires ALL sections populated: about, skills, experience, projects, achievements, certifications, contact. All data is available in the initial prompt. Fill it ALL in Phase 1.
**Warning signs:** Empty arrays or placeholder strings in `portfolio.ts`.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `tailwind.config.js` (JS config) | `@theme` directive in CSS | Tailwind v4 (Jan 2025) | All theme customization is CSS-only. No JS config file. |
| `tailwindcss-animate` plugin | `tw-animate-css` package | shadcn/ui + Tailwind v4 | Import as CSS, not as plugin. |
| `framer-motion` package | `motion` package | Rebranded 2024 | Same API, new package name. Use `motion` for new projects. |
| `next.config.js` | `next.config.ts` | Next.js 15+ | TypeScript config supported natively. |
| `darkMode: "class"` in tailwind config | `@custom-variant dark (...)` in CSS | Tailwind v4 | Dark mode configuration moved to CSS. |
| HSL color values in shadcn/ui | OKLCH color values | shadcn/ui + Tailwind v4 | Better perceptual uniformity. New installs use OKLCH. |

**Deprecated/outdated:**
- `tailwind.config.js` / `tailwind.config.ts`: Legacy in Tailwind v4. Use `@theme` in CSS.
- `tailwindcss-animate`: Deprecated for Tailwind v4. Use `tw-animate-css`.
- `@next/font`: Removed. Use `next/font/google` or `next/font/local`.
- `next-themes`: Skip for dark-only sites. Hardcode dark theme directly.

## Open Questions

1. **Geist font availability via next/font**
   - What we know: Geist is Vercel's font and ships in create-next-app defaults. It should be available via `next/font/google` or may be bundled.
   - What's unclear: Exact import path may vary between Next.js versions. create-next-app may use `next/font/local` with bundled Geist files.
   - Recommendation: Use whatever font import the scaffold generates. If Geist isn't easily available, use Inter as fallback sans-serif font. The monospace font (JetBrains Mono) is the primary font for this project anyway.

2. **shadcn/ui color variable names**
   - What we know: shadcn/ui init generates CSS variables for its component system (e.g., `--background`, `--foreground`, `--card`, etc.)
   - What's unclear: Exact variable names and format may differ between shadcn/ui versions targeting Tailwind v4.
   - Recommendation: Run `npx shadcn@latest init` and inspect generated `globals.css`. Merge shadcn's variables with our custom console theme variables. Don't fight shadcn's naming -- extend it.

3. **create-next-app in existing git repo**
   - What we know: The project directory already has `.git` and `.planning`. Running `create-next-app` typically creates a new directory.
   - What's unclear: Whether `create-next-app` can initialize in an existing directory without conflicts.
   - Recommendation: Run `create-next-app` in a temp directory, then move files into the project root. Or use `--disable-git` flag and run in the existing directory. The `--disable-git` flag prevents it from running `git init`.
